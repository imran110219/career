# Architecture Governance

Governance preparation should show practical judgment, not bureaucracy.

## Focus Areas

| Area | Practice focus |
| --- | --- |
| Architecture review | Review service boundaries, security, data ownership, deployment and operability. |
| Decision records | Capture context, decision, alternatives, trade-offs and consequences. |
| Technology standards | Define defaults for APIs, persistence, testing, observability and security. |
| Exception handling | Allow justified deviations with review date and owner. |
| Service ownership | Identify owners, runbooks, SLIs, dependencies and lifecycle. |
| Backward compatibility | Manage API and event changes without breaking consumers. |
| Technical debt | Track risk, impact, owner and planned resolution. |
| Fitness functions | Automate architecture checks where practical. |
| Cross-team alignment | Use RFCs, design reviews and lightweight standards. |
| Platform reliability | Tie decisions to operability, recovery and support burden. |
